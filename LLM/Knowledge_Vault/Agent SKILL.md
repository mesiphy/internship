# 1. Agent Skill 的核心概念与背景

**详细讲解：**

- **背景与定位：** Agent Skill 最初是 Anthropic 为了提升 Claude 在特定任务上的表现而推出的，但由于其设计通用且好用，VS Code、Cursor 等工具迅速跟进支持。2025年12月18日，它被正式发布为开放标准，演变为 AI Agent 领域通用的设计模式。
- **通俗理解：** Agent Skill 本质上是一个大模型可以随时翻阅的“说明文档”。通过在文档中预设好规则（如智能客服的安抚话术、会议总结的特定输出格式），用户就不需要每次对话都重复输入要求，大模型会自动查阅并照做。

**初学者可能存在的问题（视频未详细讲解）：**

- **环境与兼容性问题：** 视频提到使用“Cloud Code”（即 Claude Code）来演示，但没有讲解初学者如何安装、配置 Claude Code，也没有说明这个功能是否需要付费账号或特定版本的 API 支持。
- **跨平台通用性的具体体现：** 视频提到它支持跨平台复用，但初学者可能不清楚同一个 Skill 文件是如何在 VS Code 和 Cursor 中无缝切换使用的。

## 2. 基础用法与按需加载机制

**详细讲解：**

- **文件结构：** 创建一个 Skill 需要在特定的目录下（如用户目录的 `.claude/skills` 下，视频中口误为 `DCloud scale`）创建一个与 Skill 同名的文件夹（例如“会议总结助手”），并在其中创建 `skill.md` 文件。
- **skill.md 的组成：** 文件分为两部分。头部是被短横线包围的“元数据（Meta）”层，包含 `name`（名称，必须与文件夹同名）和 `description`（描述，告诉大模型这个组件是干嘛的）。下半部分是“指令（Instruction）”层，详细描述模型需要遵循的规则，还可以包含输入输出的示例（Few-shot prompting）。
- **按需加载机制（第一层与第二层）：** 视频强调了节省 Token 的“渐进式披露结构”。一开始，系统只会把所有 Skill 的元数据（名称和描述，相当于轻量级目录）发给大模型。只有当大模型判断用户请求与某个 Skill 匹配时，系统才会去读取那个特定 `skill.md` 的完整指令内容并发送给大模型，这就是“按需加载”。

**初学者可能存在的问题（视频未详细讲解）：**

- **语法严格性：** 视频提到“被短横线包起来”，这实际上是标准的 YAML Frontmatter 格式。初学者如果不了解缩进、冒号后的空格等语法细节，极容易导致解析失败。
- **多个指令冲突：** 视频没有讲解如果用户的输入同时命中了两个相似的 Agent Skill，大模型会如何选择，或者是否会报错。

### 3. 高级用法：Reference（资源引用）

**详细讲解：**

- **解决的痛点：** 如果把所有辅助资料（如冗长的财务规定、法律条文）都写进 `skill.md`，每次触发都会消耗大量 Token。
- **工作原理：** Agent Skill 提供了 Reference 功能，允许创建额外的文件（如 `集团财务手册.md`）。你可以在 `skill.md` 中写明触发规则（例如“当提到钱、预算时触发”），只有满足条件时，系统才会去读取这个 Reference 文件。
- **特性：** 这被称为“按需中的按需加载”（第三层结构）。Reference 的核心操作是**“读”**，它会被加载到模型的上下文中作为参考，因此会消耗 Token，但仅在绝对必要时才消耗。

**初学者可能存在的问题（视频未详细讲解）：**

- **触发指令的写法：** 视频没有展示具体应该用什么样的提示词（Prompt）在 `skill.md` 中才能精准触发系统去读取这个外部文件。是自然语言描述，还是特定的语法标签？
- **文件路径与格式：** 引用外部文件时，是使用相对路径还是绝对路径？除了 `.md` 文件，是否支持 PDF 或 CSV 等其他格式的文档？

### 4. 高级用法：Script（脚本执行）

**详细讲解：**

- **核心能力：** 除了查资料，Agent Skill 还能直接运行代码（如 Python 脚本）来实现真正的自动化（如自动上传文件到服务器）。
- **工作原理：** 在文件夹中创建脚本（如 `upload.py`），并在 `skill.md` 中描述触发条件（如提到“上传”、“同步”时运行此脚本）。
- **特性：** Script 的核心操作是**“跑”**。系统只关心脚本的运行方法和结果，**不会**去读取脚本里面的代码内容放到上下文中（除非运行失败需要排错），因此无论脚本逻辑多复杂，消耗的模型上下文（Token）几乎为 0。

**初学者可能存在的问题（视频未详细讲解）：**

- **参数传递机制：** 大模型在决定运行 `upload.py` 时，是如何将“会议总结的内容”传递给 Python 脚本的？是通过命令行参数（CLI args）、标准输入（stdin），还是临时文件？初学者在这里无法独立完成代码编写。
- **安全与环境问题：** 允许大模型自动执行本地计算机上的 Python 脚本存在很大的安全隐患（如误删文件）。同时，如果脚本依赖外部库（如 `requests`），如何管理 Python 环境？视频对此只字未提。

### 5. Agent Skill 与 MCP 的对比

**详细讲解：**

- **核心区别：** 视频引用了 Anthropic 官方的解释：“MCP 连接数据，而 Skills 教 Claude 如何处理这些数据”。MCP 的作用是给大模型供给外部数据（比如查询数据库、物流状态），而 Agent Skill 是教大模型如何按规矩处理这些数据（比如必须输出特定的格式）。
- **架构差异与选择：** 尽管 Agent Skill 也能写脚本连接数据，但它本质上只是一段“说明文档”，适合跑轻量级脚本和简单逻辑。相比之下，MCP 本质上是一个“独立运行的程序”，在代码执行的安全性和稳定性上远胜于 Skill。
- **最佳实践：** 就像“瑞士军刀也能切菜但没人这么干”一样，专业的场景下不应该用 Skill 去强行包揽一切，而应该将 Agent Skill（负责逻辑规则）和 MCP（负责数据连接）结合起来使用。

**初学者可能存在的问题（视频未详细讲解）：**

- **结合使用的具体演示：** 视频在最后抛出了“两者需要结合使用”的观点，但并没有给出任何代码演示或架构图。初学者依然不知道如何让一个 Agent Skill 内部去调用一个具体的 MCP Server 获取数据。
- **MCP 本身的门槛：** 视频假设观众已经知道 MCP 是什么，初学者如果没接触过 MCP（Model Context Protocol），会很难理解“提供数据”与“处理数据”在代码工程层面究竟是如何隔离的。
