# LLM Harness Engineering

## Prompt Engineering

提示词的本质，是塑造一个局部概率空间，让大模型这个“概率生成模型”生成我们想要的输出的概率更高。

对应的问题是“表达的问题”，目的是“让大模型*理解*你的问题”

使用场景：
- 澄清任务
- 约束输出
- 激发模型已有能力

## Context Engineering

系统在合适的时机把正确的信息告知模型，context代表了所有会影响系统当前输出结果的信息综合。

目的是在项目推进的整条链路上打通模型获取信息的渠道：
- 文档切块、结果排序、长文压缩、历史对话的处理
- 工具返回结果处理、Agent间传递信息的方法

典型实践：
- RAG
- SKILLS:渐进式批漏信息给大模型，避免注意力涣散的问题。

使用场景：
- 按需给
- 分层给
- 在合适的时机给

## Harness Engineering

提示词、上下文工程都是在 <font color="#f79646">信息供给侧</font> 优化，现在在Agent为主力的工作流程中，需要解决的问题是：谁来<font color="#f79646">监督和约束</font>每一步输出结果。

目的：别跑偏、跑的稳、出错及时发现并纠正

<font color="#f79646">Harness = Agent - Model </font>

除了model之外任何保证Agent输出结果的部分都属于Harness(缰绳)

### 第一职责：让模型在边界中思考

组成：
- 角色和目标定义
- 信息选择和裁剪：信息越相关越好
- 结构化的组织

### 工具系统

聚焦的问题：
- 给什么工具
- 什么时候该调用工具
- 结果怎么重新喂回：将工具返回的结果进行筛选

### 完整的Agent工作流程

1. **理解目标**：明确当前需要完成的任务是什么。
2. **判断与补充信息**：判断当前拥有的信息够不够，如果不够就需要继续补充。
3. **分析**：结合补充后的结果继续进行分析。
4. **生成输出**：根据分析的结果生成相应的内容或操作。
5. **检查与修正**：检查输出的内容，如果不满足要求，就重新修正或者重试。

这套流程使得Agent的表现非常接近人类在工作，区别在于人类依靠经验，而Agent依靠Harness这套运行环境。

### Harness Engineering的六层架构

在Agent系统中，除了模型本身以外，几乎所有能决定它能否稳定交付的东西都可以算作Harness（Agent = Model + Harness）。一个成熟的Harness Engineering可分为六层：

1. **重新审视上下文（信息边界）**：让模型在正确的信息边界内思考。包括：明确角色的目标和定义；信息的裁剪和选择（越相关越好）；结构化地组织信息（固定规则、当前任务、外部证据分层存放，防止污染）。
2. **工具系统**：模型连上工具才能真正做事。需要解决：给什么工具（避免太少或太多乱用）、什么时候调用（该查的时候查，不该查的时候别硬答）、工具结果如何重新喂给模型（提炼筛选后再给模型）。
3. **执行编排**：解决模型下一步该做什么的问题，把所有步骤串起来形成轨道。例如：理解目标 -> 寻找信息 -> 分析 -> 生成 -> 检查 -> 修正。
4. **记忆与状态**：管理Agent的状态，避免其像失忆一样。必须分清三类东西：当前任务的状态、会话中的中间结果、长期记忆和用户偏好。
5. **评估与观测**：解决系统"不知道自己做得好不好"的问题。包括：输出和验收、环境验证、自动测试、日志与指标、错误归因。
6. **约束、校验、失败与恢复**：决定系统能否上线的关键环节。包含：约束（哪些不能做）、校验（输出前后怎么检查）、恢复（失败后怎么重试、切换路径或回滚到稳定状态）。
